# ArcFlow 前半段主流程测试文档

- 文档日期：2026-04-17
- 适用范围：仅验证前半段主流程，不覆盖 `code_gen`、CI、Bug 回流
- 目标链路：`需求对话/草稿 -> finalize -> 飞书 Review -> Approve -> Plane Issue + PRD Git -> Plane webhook -> 技术设计 + OpenAPI`

## 1. 测试目标

确认当前 ArcFlow 前半段主流程可以从 Web 侧发起，并在外部系统协同后形成完整产物：

1. 能在 ArcFlow Web 中发起需求对话或生成需求草稿
2. 能完成 `finalize`，进入评审状态
3. 能在飞书中收到评审卡片或审批入口
4. 能完成 Plane Issue 与 PRD 文档落库
5. 能由 Plane webhook 触发后续 AI 处理
6. 能生成技术设计文档与 OpenAPI 文档

## 2. 不在本次范围内

以下内容不作为本次测试通过条件：

- `code_gen`
- GitHub PR / Code Review
- CI 成功或失败回流
- `bug_analysis`
- 自动修复闭环

## 3. 测试前提

执行测试前，请先确认下面条件成立。

### 3.1 服务状态

至少应有以下服务可用：

- ArcFlow Web
- ArcFlow Gateway
- NanoClaw
- Plane CE
- docs Git 仓库
- 飞书应用配置

建议至少验证以下地址或等价入口：

- ArcFlow Web 可正常打开并登录
- Gateway 健康检查可访问，例如 `/health`
- Plane 页面可访问
- NanoClaw 进程在线

### 3.2 关键配置

以下配置应已正确设置，否则前半段很容易卡在 webhook 或审批环节：

- 飞书 OAuth 配置可用
- 飞书消息/卡片发送配置可用
- Plane API Token 可用
- Plane 项目已与 ArcFlow workspace 建立映射
- Plane Approved 状态 ID 已配置
- Plane webhook 已指向 Gateway
- docs 仓库地址已绑定到当前 workspace
- Gateway 已配置 NanoClaw 地址和派发密钥

### 3.3 测试账号

建议准备以下账号或权限：

- 一个可登录 ArcFlow Web 的飞书账号
- 一个可在 Plane 中查看并操作目标项目 Issue 的账号
- 一个能查看飞书消息/卡片的账号
- 一个可查看 docs Git 仓库提交记录的账号

## 4. 测试数据建议

为了便于确认产物是否来自本次测试，建议使用一个明显可识别的需求标题。

建议格式：

- 标题：`前半段联调测试-YYYYMMDD-HHMM`
- 内容中加入唯一标记，例如：`TEST-RUN-20260417-01`

建议测试需求示例：

> 为工作流执行列表增加“仅看我负责的需求”筛选能力，支持按状态和负责人组合筛选，并在详情页显示筛选条件摘要。

## 5. 测试步骤

## 5.1 登录 ArcFlow 并确认工作空间

操作：

1. 打开 ArcFlow Web
2. 使用测试账号登录
3. 切换到目标 workspace
4. 进入 AiChat 或需求草稿入口

预期结果：

- 页面可正常加载
- 当前 workspace 正确
- 无 401、403、空白页或明显前端报错

失败时优先检查：

- 登录态是否失效
- 当前账号是否属于该 workspace
- 浏览器开发者工具中的接口报错

## 5.2 发起需求对话或创建需求草稿

操作：

1. 在 AiChat 中输入测试需求
2. 如果当前流程需要多轮补充，则继续补充上下文
3. 等待系统返回可用于生成 PRD 的内容

建议记录：

- 输入的原始需求
- 是否出现结构化 artifact 卡片
- AI 返回是否包含明显的 PRD 草稿内容

预期结果：

- 可以正常发送消息
- 能收到 AI 返回
- 返回内容能形成一个明确需求，不是空结果或报错

失败时优先检查：

- NanoClaw 是否在线
- Gateway 到 NanoClaw 的调用是否正常
- AiChat 页面是否有 SSE 中断或超时

## 5.3 执行 finalize

操作：

1. 在当前需求草稿或对话中执行 `finalize`
2. 等待系统完成状态更新

预期结果：

- 需求进入 `review` 或等价待评审状态
- 前端出现“已提交评审”或等价反馈
- 不出现 401、500 或超时

建议记录：

- 对应草稿 ID 或会话 ID
- finalize 提交时间

失败时优先检查：

- Gateway 日志中是否有 finalize 处理报错
- 飞书通知逻辑是否报错
- 当前 workspace 的 docs 仓库绑定是否正确

## 5.4 检查飞书评审卡片或审批入口

操作：

1. 打开飞书，进入目标通知群或接收消息的会话
2. 查找本次测试对应的评审卡片或审批入口

预期结果：

- 能收到与本次测试需求对应的飞书消息
- 卡片标题、摘要或链接能识别出本次测试内容

失败时优先检查：

- 飞书 chat_id / webhook / app 配置
- Gateway 发送通知日志
- finalize 是否真的进入评审阶段

## 5.5 在飞书或相关入口执行通过

操作：

1. 按当前实际流程执行“通过评审”
2. 如果是跳转 ArcFlow/Plane 页面后再确认，也记录实际路径

预期结果：

- 系统接受审批动作
- 不出现无权限、链接失效或回调失败

建议记录：

- 审批动作发生时间
- 使用的是飞书卡片按钮、Web 跳转页还是其他入口

失败时优先检查：

- 审批 token 或链接是否过期
- 飞书回调是否按当前方案配置
- ArcFlow/Gateway 是否可被浏览器正常访问

## 5.6 检查 Plane Issue 与 PRD 文档是否落库

操作：

1. 打开目标 Plane 项目
2. 查找本次测试生成的 Issue
3. 打开 docs 仓库，查看是否生成对应 PRD 文档
4. 查看 docs 仓库最近提交记录

预期结果：

- Plane 中出现本次测试对应的 Issue
- Issue 标题、描述与测试需求基本一致
- docs 仓库出现新的 PRD 文档
- docs 仓库存在对应 commit

建议重点核对：

- Issue 是否关联到正确项目
- PRD 路径是否符合仓库约定
- 文档内容是否与本次测试唯一标记一致

失败时优先检查：

- Plane API token
- workspace 与 Plane project 的映射关系
- docs 仓库写入权限
- Gateway 原子写入逻辑日志

## 5.7 检查 Plane 状态流转与 webhook 触发

操作：

1. 在 Plane 中确认 Issue 已进入 Approved 或等价目标状态
2. 查看 Gateway 日志或工作流记录
3. 确认 webhook 到达 Gateway

预期结果：

- Gateway 收到来自 Plane 的 webhook
- 产生与本次测试相关的工作流执行记录
- 该执行记录与当前 Plane Issue 对应

优先检查入口：

- ArcFlow Web 的工作流列表/详情页
- Gateway webhook 日志接口或服务日志

失败时优先检查：

- Plane webhook URL 是否正确
- webhook secret / header 校验
- Approved 状态 ID 是否与当前 Plane 配置一致

## 5.8 检查技术设计与 OpenAPI 产物

操作：

1. 等待后续 AI 流程完成
2. 在 docs 仓库检查是否新增技术设计文档
3. 在 docs 仓库检查是否新增 OpenAPI 文档
4. 查看 ArcFlow 工作流详情，确认相关执行状态

预期结果：

- 生成技术设计文档
- 生成 OpenAPI 文档
- 两者都能在 docs 仓库中找到
- 工作流状态为成功，或至少能明确卡住的阶段

建议核对：

- 技术设计文档路径
- OpenAPI 文档路径
- 文档内容是否与本次测试需求一致

失败时优先检查：

- NanoClaw skill 是否正确执行
- Gateway callback 是否收到结果
- docs 写入是否成功
- 工作流详情中的错误信息、子任务和关联记录

## 6. 通过标准

满足以下条件即可认为“前半段主流程测试通过”：

1. 能从 ArcFlow Web 发起有效需求输入
2. 能成功执行 finalize
3. 能收到飞书评审消息或审批入口
4. 能创建 Plane Issue
5. 能在 docs 仓库生成 PRD 文档
6. Plane 审批后能触发后续流程
7. 能在 docs 仓库生成技术设计文档
8. 能在 docs 仓库生成 OpenAPI 文档

如果只有前 5 项成功，而技术设计/OpenAPI 未生成，则应判定为：

- 前半段前置链路成功
- webhook 之后的 AI 处理链路失败

## 7. 建议排查顺序

测试失败时，建议按这个顺序排查：

1. Web 页面是否报错
2. Gateway 是否收到对应请求
3. 飞书通知是否发出
4. Plane Issue 是否创建
5. Plane webhook 是否到达 Gateway
6. NanoClaw 是否收到 dispatch
7. Gateway callback 是否回写成功
8. docs 仓库是否真正写入产物
